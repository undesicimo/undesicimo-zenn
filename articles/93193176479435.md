## はじめに
2026年4月から社会人5年目に入るので、ここでいったんこの4年間を振り返ってみます。
この4年間を振り返ってみると、自分のキャリアの軸はかなり一貫していたのではないかと思います。

もともと自分はパソコンが好きで、ゲームもいっぱいやってました。英語も好きで、アメリカ文化やゲームを通じて自然と触れることが多く、その流れで大学では外国語学部英語学科を専攻しました。

当時は、今のようなエンジニアリングの仕事に関わるようになるとはほとんど想像していませんでした。

今振り返ると、好きだったものが全部あとからつながってきた、という感じがします。

## テストエンジニアとして働き始めたころ(1年目〜)

新卒で入社して最初にやっていたのは、顧客先に常駐する形でテストエンジニアとして業務を進める仕事でした。

いわゆる「依頼されたことを確実にこなす」タイプの働き方で、日々の業務の中では手動で API テストを進めることもありました。

この手動 API テストが、かなり大変でした。

- 手間がかかる
- 同じことを何度も繰り返す
- やればやるほど気が重くなる

このあたりで初めて、「これって自動化できないのか？」と思うようになりました。

そして調べていく中で、テスト自動化という世界があること、その延長線上に自動化エンジニアというキャリアがあることを知りました。

今思えば、このタイミングが自分にとって最初の大きな分岐点でした。

## 独学でテスト自動化に触れた

テストエンジニアとして働きながら、Udemy で Java と Selenium を使ったテスト自動化の講座を受講しました。

実際に手を動かしてみると、これがかなりおもしろかったです。

自分で書いたものが勝手に動いて、繰り返しの作業を肩代わりしてくれる。この感覚にかなり強く惹かれました。

「あ、自分はこれが好きなんだな」とはっきり自覚したのは、この頃だったと思います。

一方で、興味を持ったからといって、すぐに仕事で実践できるわけではありませんでした。

やりたいことは見えているのに、今いる環境ではまだそこに踏み込めない。そのもどかしさはずっとありました。

## やりたいことと、すぐにはできない現実

当時の現場では、テスト自動化を提案してみたこともありました。

ただ、相手は大手スマホアプリ会社で、受託側の立場では「求められたことをやる」以上の動きをするのが難しく、実現には至りませんでした。

その後、自社の別案件で自動化を身につけられないかも相談しましたが、当時の状況ではすぐに異動するのも難しく、現実的には独学を続けるしかありませんでした。

この時期は、進みたい方向はかなり見えてきているのに、環境の都合ですぐにはそちらに行けない、という感覚が強かったです。

ただ、今振り返ると、この期間に手を止めずに学び続けていたことがかなり大きかったと思います。

## ビットキーで SET に挑戦した(2年目〜3年目)

そんな中で見つけたのが、テスト自動化エンジニア（SET）の募集でした。

応募してジョインした頃は、ちょうどチームができて間もないタイミングでした。自分自身は完全に実務未経験の初心者としてのスタートで、メンバーもまだ少なく、これから作っていく余地の大きい環境だったと思います。

ジョイン後は、メンターとプログラミング研修カリキュラムが用意されていました。

最初は JavaScript で書く FizzBuzz から始まったのですが、正直かなり苦戦しました。

- レビューはぼろぼろ
- ビルド後のコードまで push してしまう
- プリミティブな値の違いもよく分からない
- `var`、`let`、`const` の違いも理解できていない

当時の Pull Request を今見返すと、かなり初心者らしさが出ています。

[https://github.com/undesicimo/nodejs-typescript-fizzbuzz/pull/4](https://github.com/undesicimo/nodejs-typescript-fizzbuzz/pull/4)

ただ、この時期にちゃんと打ちのめされたのは良かったと思っています。

自分がどこでつまずくのか、何が分かっていないのかをかなり具体的に認識できましたし、そこから少しずつ積み上げていくしかないと腹をくくれました。

同時に、完全にプログラミング未経験の初心者だった自分に対して、学ぶ機会を用意してくれて、根気強くレビューやフィードバックをしてくれたメンターや周囲の人たちには本当に感謝しています。

当時は余裕がなくて必死でしたが、今振り返ると、あのタイミングで基礎的な部分からきちんと向き合えたことが、その後の自分をかなり支えてくれました。

その後は Express で簡単な REST API サービスを実装したり、初めて React を書いたりしながら、少しずつできることを増やしていきました。React については、公式の [Tutorial: Tic-Tac-Toe – React](https://react.dev/learn/tutorial-tic-tac-toe) をベースに学んだことをよく覚えています。

## コードを書く以外にも、品質のための武器が増えた

この4年間で身についたのは、単にコードを書く力だけではありませんでした。

品質に関わる立場として、さまざまなツールや考え方に触れられたのも大きかったです。

たとえば、研修を終えてからは React Testing Library や Jest を使って単体テストをいっぱい書きました。主要なバックエンドやフロントエンドのカバレッジを上げたり、VRT の基盤を整えたり、フロントエンドのマイグレーションのような責任の大きいプロジェクトにも関わるようになっていきました。

こういう経験を積む中で改めて感じたのは、**やっぱり自分はテスト自動化が好きなんだ**ということです。

単に手作業を減らしたいというだけではなく、繰り返しを仕組みに変えて、人が本当に考えるべきところに時間を使えるようにする。この感覚に自分はずっとおもしろさを感じています。

## 仕事の外でも作ることを続けた

一方で、仕事の外でも何かを作ることは続けていました。

Web デザイナーをしている妻と一緒に、個人用の Web アプリやホームページを作ったり、知人づてにホームページ制作の案件をこなしたりもしていました。

その流れで、ホームページで使う CMS を個人開発することもありました。

今振り返ると、こういう個人開発の時間はかなり大きかったと思います。

実務とはまた違う形で、自分で作るものを決めて、自分で責任を持って最後まで形にする経験ができたからです。業務の中で得た知識とつながることも多く、逆に個人開発で試したことが仕事に返ってくることも多かったです。

## マネーフォワードに転職してから(2年目〜4年目)

その後、現在のマネーフォワードに転職しました。

ここではこれまで以上に、品質や開発基盤に対してより広い視点で関わるようになったと思います。

### 関わる範囲が広がった

たとえば、静的コード解析ツールの導入やセキュアコーディングへの取り組みなどに関わってきました。単にテストを書くというよりも、開発全体の品質をどう底上げするかを考える機会が増えた感覚があります。

### リードする立場になってきた

さらに、社内向けの API テストフレームワークや QA 向けの AI ツールの開発を、自分がリードする立場で進める機会も増えていきました。

最初は学ぶ側として入った自分が、少しずつ仕組みを作る側になり、さらにそれをリードする側になっていったのは、この数年の変化のひとつでした。

### 使いやすさを意識するようになった

その中で意識するようになったのが、昔の自分のような人でも使いやすいものにすることです。

自分は、手作業のつらさは分かっていても、そこからどう自動化に入ればいいのか分からない、という状態を実際に経験してきました。

だから今ツールを作るときは、機能の多さだけではなく、最初の一歩を踏み出しやすいこと、迷いにくいこと、前提知識がなくても扱いやすいことを重視するようになりました。

高機能でも、導入や理解のハードルが高いと、実際には使われにくいことが多いと感じています。だからこそ、自分が作るものは、まず使われることと、続けて使えることが大事だと思っています。

昔につまずいた経験や、基礎から学んだ時間は、今こういう判断をする上で役に立っています。AI が当たり前になった今でも、その土台があることで見え方はかなり変わると感じています。

## 最後に

まだ書き切れていないことはかなりありますが、この4年間を振り返ってみると、興味を持ったことに対して、すぐに環境が追いつかない時期があっても手を止めずに続けてきたことが、少しずつ今につながっているのだと思います。

最初は「手動作業がつらい」というところから始まった関心でしたが、そこから自動化に出会い、学び、実務で挑戦し、品質という領域そのものの見え方も少しずつ変わってきました。

この先も、エンジニアリングそのものや、テスト自動化・品質の仕事を通じて、より良い開発の進め方を作っていきたいと思います。
